# 동아대학교 승학캠퍼스 교내 길찾기

## 1. 프로그램 소개

`navDay02.py`는 동아대학교 승학캠퍼스 안에서 출발지, 경유지, 도착지를 선택해 이동 경로를 찾는 로컬 웹 프로그램이다.

사용자가 대표 건물이나 버스 정류장을 입력하면 `navDay02.sql`의 장소 및 도로 데이터를 그래프로 구성하고 최단 경로를 계산한다. 계산된 경로는 카카오 지도 위에 선과 순서 번호로 표시되며, 화면 왼쪽에는 총 거리, 예상 시간, 계단 여부, 실내 여부 등의 구간 정보가 나타난다.

별도의 웹 프레임워크 없이 Python 표준 라이브러리의 HTTP 서버를 사용하며, 기본 접속 주소는 다음과 같다.

```text
http://127.0.0.1:8000
```

## 2. 주요 기능

- 출발지와 도착지 검색
- 출발지와 도착지 맞바꾸기
- 여러 경유지 추가 및 삭제
- 브라우저 GPS를 이용한 현위치 출발 또는 도착
- 거리 우선 또는 시간 우선 경로 검색
- 차량 또는 도보 이동 방식 선택
- 계단 최소화 및 실내 위주 옵션
- 카카오 지도 일반 지도 및 위성 지도 전환
- 경로를 지도 위 선과 순서 번호로 표시
- 구간별 거리, 시간, 계단 여부, 실내 여부 표시
- 입력 영역과 경로 정보 영역 접기 및 펼치기
- 출발지와 도착지 바로 아래에 경유지 입력란 추가
- 입력 영역을 접을 때 제목과 설명을 포함한 전체 입력 UI 숨김

## 3. 프로그램 구성

| 파일 | 역할 |
| --- | --- |
| `navDay02.py` | 데이터 로딩, 경로 탐색, HTTP API, 웹 화면을 모두 담당하는 실행 파일 |
| `navDay02.sql` | 장소와 도로의 초기 데이터를 저장한 SQL 스크립트 |
| `navDay02.md` | 프로그램과 데이터 구조를 설명하는 문서 |
| `.env` | 로컬에서 사용할 카카오 JavaScript 키를 저장하는 Git 제외 파일 |

`navDay02.py`는 현재 MySQL 서버에 직접 접속하지 않는다. 프로그램을 시작할 때 `navDay02.sql`에서 `place_data`와 `road_data`의 `INSERT` 문을 읽어 Python 객체로 변환한 뒤 메모리에서 사용한다.

따라서 SQL 파일을 수정한 경우 실행 중인 서버를 종료하고 다시 시작해야 변경된 데이터가 반영된다.

## 4. 데이터 구성

`navDay02.sql`은 `데이터베이스 설정.xlsx`의 `place_data`, `road_data` 시트를 바탕으로 생성된 초기 데이터 스크립트다. 이 SQL 파일은 `nav` 데이터베이스와 두 테이블을 생성한 뒤 장소 및 도로 데이터를 입력하도록 구성되어 있다.

현재 SQL 데이터는 다음과 같이 구성되어 있다.

| 구분 | 개수 |
| --- | ---: |
| 전체 장소 | 130개 |
| 도로 연결 | 30개 |
| 검색 가능한 대표 장소 | 28개 |
| 경로에 실제로 연결된 장소 | 30개 |

### 4.1 place_data

`place_data`는 지도에 표시하거나 경로의 꼭짓점으로 사용하는 장소 정보를 저장한다.

| 컬럼 | 자료형 | 설명 |
| --- | --- | --- |
| `place_id` | VARCHAR | 장소를 구분하는 코드 |
| `place_name` | VARCHAR | 사용자에게 보여 줄 장소 이름 |
| `latitude` | DOUBLE | 위도 |
| `longitude` | DOUBLE | 경도 |
| `altitude` | DOUBLE | 고도, 값이 없으면 NULL |

장소 코드는 역할에 따라 구분된다.

| 코드 형식 | 의미 | 예시 |
| --- | --- | --- |
| `GATE` | 캠퍼스 정문 | `GATE` |
| `S01` ~ `S31` | 사용자가 선택하는 대표 건물 | `S06` |
| `BSxx` | 버스 정류장 | `BS06` |
| `Sxx-...` | 건물의 실제 출입구 또는 층별 연결 지점 | `S06-06F03E` |
| `CRxx` | 도로의 교차점 또는 경로 연결 지점 | `CR09` |
| `PLxx` | 주차장 또는 도로 연결 지점 | `PL04` |

기본 지도와 검색 목록에는 대표 장소인 `GATE`, `S01`부터 `S31`, `BSxx`만 표시한다. 출입구, 교차점, 주차장 코드는 사용자가 직접 목적지로 선택하지 않고 경로를 계산하고 지도에 그릴 때 사용한다.

현재 검색 가능한 대표 장소는 다음과 같다.

```text
GATE
S01 ~ S20, S31
BS01, BS04, BS06, BS11, BS15, BS31
```

### 4.2 road_data

`road_data`는 두 장소 사이의 이동 가능한 연결을 저장한다. 경로 탐색에서는 각 행을 그래프의 간선으로 사용한다.

| 컬럼 | 자료형 | 설명 |
| --- | --- | --- |
| `road_id` | INT | 도로 데이터의 고유 번호 |
| `from_place` | VARCHAR | 출발 장소 코드 |
| `to_place` | VARCHAR | 도착 장소 코드 |
| `road_type` | VARCHAR | 차도, 보행로 등 도로 종류 |
| `distance` | INT | 구간 거리, 단위는 m |
| `time` | INT | 예상 이동 시간, 단위는 분 |
| `indoor` | BOOLEAN | 실내 구간이면 1, 실외면 0 |
| `stair` | BOOLEAN | 계단이 있으면 1, 없으면 0 |
| `slope` | INT | 경사도 또는 경사 비용 |

예를 들어 다음 데이터는 정문에서 `CR01-01`까지 203m이며 예상 시간이 3분인 실외 차도를 의미한다.

```sql
(1, 'GATE', 'CR01-01', '차도', 203, 3, 0, 0, NULL)
```

SQL에는 한 방향으로 저장되어 있어도 현재 프로그램은 모든 도로를 양방향으로 연결한다. 따라서 `GATE → CR01-01` 데이터가 있으면 반대 방향인 `CR01-01 → GATE`도 이동할 수 있다고 처리한다.

## 5. 대표 건물과 출입구 연결

대표 건물 코드는 검색에 사용되고, 실제 경로는 도로에 연결된 출입구 코드를 사용한다.

예를 들어 사용자가 `S06`을 선택하면 프로그램은 도로 데이터에 직접 연결된 다음과 같은 `S06-` 접두사의 출입구를 후보로 찾는다.

```text
S06-06F01E
S06-06F03E
```

여러 출입구가 있으면 각 출입구를 기준으로 경로를 비교한 뒤 비용이 가장 낮은 경로를 선택한다. 경로 목록의 출발지, 경유지, 도착지에는 내부 출입구 코드 대신 사용자가 선택한 대표 건물 정보를 표시한다.

## 6. 경로 탐색 방식

프로그램은 장소를 정점, 도로를 간선으로 하는 그래프를 만들고 다익스트라 알고리즘으로 최단 경로를 계산한다.

경로 계산 순서는 다음과 같다.

1. 출발지, 경유지, 도착지 코드를 정규화한다.
2. 대표 건물에 연결된 실제 출입구 후보를 찾는다.
3. `road_data`로 양방향 그래프를 만든다.
4. 거리 또는 시간을 비용으로 사용해 가장 비용이 낮은 경로를 찾는다.
5. 경유지가 있으면 출발지부터 각 경유지와 도착지까지 구간별로 탐색한다.
6. 각 구간의 경로, 거리, 시간을 합쳐 하나의 결과로 반환한다.
7. 결과를 경로 목록과 카카오 지도 위에 표시한다.

### 경로 옵션의 비용 계산

- 거리 우선: `distance` 값을 기본 비용으로 사용한다.
- 시간 우선: `time` 값을 기본 비용으로 사용한다.
- 계단 최소화: 계단 구간에 매우 큰 추가 비용을 적용한다.
- 실내 위주: 실외 구간에 추가 비용을 적용한다.
- 경사도: 값이 있으면 기본 비용에 경사 비용을 더한다.

## 7. 현위치 처리

브라우저의 Geolocation API로 사용자의 위도와 경도를 가져온다. 현위치는 실제 SQL 장소가 아니며 프로그램 내부에서 `CURRENT_LOCATION`이라는 임시 코드로 처리한다.

현위치를 경로에 사용하면 현재 좌표에서 가까운 연결 장소 5개를 찾고, 그중 전체 비용이 가장 낮은 연결점을 선택한다. 현재 좌표와 첫 연결점 사이의 거리는 두 좌표의 직선거리를 계산하는 하버사인 공식으로 구한다.

현위치에서 도로 연결점까지의 첫 구간은 `road_data`에 없는 임시 직선 연결이다. 따라서 실제 보행로를 정확히 따르는 경로가 필요하면 현위치 주변의 보행로 데이터를 별도로 추가해야 한다.

## 8. 웹 API

Python 서버는 화면에서 사용할 데이터를 JSON으로 제공한다.

| 주소 | 설명 |
| --- | --- |
| `/` | 교내 길찾기 웹 화면 |
| `/api/places` | 검색 가능한 대표 건물과 정류장 목록 |
| `/api/map-places` | 출입구와 교차점을 포함한 전체 장소 목록 |
| `/api/roads` | 전체 도로 연결 목록 |
| `/api/route` | 조건에 맞는 경로 탐색 결과 |

`/api/route`에서 사용하는 주요 파라미터는 다음과 같다.

| 파라미터 | 값 | 설명 |
| --- | --- | --- |
| `start` | 장소 코드 | 출발지 |
| `end` | 장소 코드 | 도착지 |
| `waypoints` | 쉼표로 구분한 코드 | 경유지 목록 |
| `weight` | `distance`, `time` | 탐색 기준 |
| `travel_mode` | `car`, `walk` | 이동 방식 |
| `avoid_stairs` | `0`, `1` | 계단 최소화 여부 |
| `rainy` | `0`, `1` | 실내 위주 여부 |
| `current_lat` | 실수 | 현위치 위도 |
| `current_lng` | 실수 | 현위치 경도 |

## 9. 카카오 지도 표시

카카오 지도 JavaScript API를 사용한다. 기본 화면에서는 대표 건물과 버스 정류장만 마커로 표시한다. 경로를 검색하면 일반 마커를 숨기고 다음 정보를 표시한다.

- 이동 경로를 주황색 선으로 표시
- 경로의 각 장소를 이동 순서에 맞는 숫자로 표시
- 동일한 좌표에 여러 순서가 겹치면 번호 위치를 좌우로 조정
- 경로 전체가 화면에 들어오도록 지도 범위를 자동 조정
- 카카오 지도 기본 컨트롤로 일반 지도와 위성 지도 전환

## 10. 실행 방법

### 10.1 API 키 설정

`Chapter02-Day04/.env` 파일에 카카오 지도 JavaScript 키를 저장한다.

```text
KAKAO_MAP_API_KEY=카카오_JavaScript_키
```

프로그램은 먼저 Windows 또는 터미널의 `KAKAO_MAP_API_KEY` 환경변수를 확인하고, 값이 없으면 실행 파일과 같은 폴더의 `.env`를 읽는다. `.env`는 `.gitignore`에 등록되어 GitHub에 올라가지 않는다.

환경변수를 직접 사용하는 방법도 지원한다.

```powershell
$env:KAKAO_MAP_API_KEY="카카오_JavaScript_키"
python Chapter02-Day04/navDay02.py
```

### 10.2 서버 실행

프로젝트 폴더에서 다음 명령을 실행하거나 VS Code에서 `navDay02.py`를 F5로 실행한다.

```powershell
python Chapter02-Day04/navDay02.py
```

브라우저에서 다음 주소로 접속한다.

```text
http://127.0.0.1:8000
```

### 10.3 서버 종료

서버를 종료하려면 실행 중인 터미널에서 `Ctrl+C`를 누른다.

서버는 Windows에서 같은 8000번 포트에 중복 실행되지 않도록 설정되어 있다. 이미 서버가 실행 중인 상태에서 다시 실행하면 포트 사용 오류가 발생하므로 기존 서버를 먼저 종료해야 한다.

`Ctrl+C`, VS Code 실행 중지, 실행 터미널 종료로 부모 프로세스가 끝나면 서버도 자동으로 종료되고 8000번 포트를 닫는다. 단순히 편집기 탭만 닫는 것은 실행 종료가 아니므로 서버가 계속 동작한다.

카카오 개발자 콘솔의 Web 플랫폼에는 실제 접속 주소를 등록해야 한다. `localhost`와 `127.0.0.1`은 서로 다른 주소로 처리되므로 프로그램 주소에 맞춰 등록한다.

### API 키 관리

카카오 지도 API 키는 Python 파일이나 Git에 직접 기록하지 않고 `KAKAO_MAP_API_KEY` 환경변수 또는 로컬 `.env`로 전달한다. 프로젝트의 `.gitignore`에는 `.env`, `.env.*`가 등록되어 있으므로 로컬 키 파일은 Git 추적 대상에서 제외된다.

커밋하기 전에는 다음 명령으로 실제 키가 변경 파일에 포함되지 않았는지 확인하는 것이 좋다.

```powershell
git diff --cached
git status
```

카카오 JavaScript 키는 지도를 불러오기 위해 브라우저로 전달되므로 개발자 도구에서 확인될 수 있다. 카카오 개발자 콘솔에서 허용 Web 도메인을 실제 서비스 주소로 제한해야 한다. 키가 GitHub나 공개 문서에 노출된 경우에는 해당 키를 재발급한 뒤 기존 키를 더 이상 사용하지 않는다.

## 11. 현재 데이터의 제한 사항

현재 프로그램에는 차량, 도보, 계단 최소화, 실내 위주 기능의 UI와 계산 구조가 준비되어 있지만 도로 데이터가 충분하지 않다.

- 등록된 도로 30개가 모두 `차도`이다.
- 모든 도로의 `indoor` 값이 0이므로 현재는 전부 실외 구간이다.
- 모든 도로의 `stair` 값이 0이므로 계단 최소화 옵션으로 제외할 구간이 없다.
- `slope` 값이 모두 NULL이므로 경사도 차이가 반영되지 않는다.
- 현재 `travel_mode` 값은 전달되지만 도로를 차량용과 도보용으로 구분하여 필터링하지 않는다.
- 일부 장소는 `road_data`에 연결되지 않아 경로를 찾을 수 없다.
- 지도에 그리는 선은 등록된 장소 좌표를 직선으로 연결하므로 실제 지도 도로의 굴곡과 완전히 일치하지 않을 수 있다.

차량과 도보 경로를 실제로 구분하려면 `road_type` 또는 별도의 이동 가능 속성을 추가하고, 보행로·계단·실내 통로 데이터를 `road_data`에 보강해야 한다. 실제 도로 모양을 따라 선을 표시하려면 각 도로 구간의 중간 좌표나 별도의 경로 좌표 데이터도 필요하다.
